# 2026-04-17 DXAX 수업 한페이지 요약

## 수업 전체 요약
이날 수업의 핵심은 **Excel과 프로젝트 관리 데이터를 AI가 읽기 좋은 형태로 구조화하고, 분석 결과를 실무형 원페이지 보고서로 바꾸는 방법**, 그리고 **그 결과를 장기적으로 재사용하기 위한 LLM 위키·메모리 구조**를 함께 익히는 것이었다. 오전에는 2주차 실무 활용 과정을 정리하며 데이터 중심 사고와 업무 데이터 관리 원칙을 다시 짚었고, 이어서 공개 데이터셋과 실무형 Excel 데이터를 활용해 AI 친화적인 표 구조, EDA 질문법, 프로젝트 관리 데이터 분석 흐름을 실습했다. 오후에는 SCQA 구조의 인사이트 보고서를 만들고, Obsidian·LLM 위키·메모리 고도화 개념까지 연결하면서 다음 주 자동화 실습의 기반을 마련했다.

## 무엇을 배웠나
- 2주차 실무 활용 구간을 정리하며, 지금까지 다룬 기획서 작성, 요약, 조사, 위키 LLM, Excel 분석이 모두 실제 업무 적용을 위한 연결된 흐름이라는 점을 복기했다.
- DX·AX의 본질은 모델 이름보다 `데이터를 다루는 능력`에 있으며, 업무 데이터를 모으고 관리하는 체계 자체가 경쟁력이라는 점을 다시 강조했다.
- 실습용 데이터 소스로 인터넷 공개 데이터, Orange3 예제 데이터, Kaggle 데이터셋을 소개했고, 실제 업무 데이터 적용 시에는 보안과 관리 기준을 함께 설계해야 한다고 설명했다.
- Excel 분석에서는 사람이 보기 좋은 보고용 시트와 AI가 읽기 좋은 분석용 시트가 다르다는 점을 구분했다. 병합 셀, 다중 헤더, 빈칸이 많은 표는 AI 해석에 불리하며, 단일 헤더와 열 단위 속성 구조가 더 적합하다고 설명했다.
- 너무 복잡한 시트는 원본 Excel을 그대로 넣기보다 이미지로 바꾸어 맥락을 유지한 채 읽히게 하는 편이 더 정확할 수 있다는 점도 시연했다.
- 분석 실습에서는 먼저 비즈니스 질문, 비교 대상, 기대 출력 포맷을 정한 뒤 Codex에 결측치·이상치 처리와 보고서 생성을 요청하는 흐름을 연습했다.
- 프로젝트 관리 데이터 약 1,000건을 가지고 연도별, 월별, 담당자별 비교를 수행하며 단순 평균보다 `상태 구성`, `진행 중 backlog`, `완료율 해석`이 중요하다는 점을 확인했다.
- EDA에서는 이상치를 무조건 제거하는 것이 아니라 `에러인지 시그널인지` 먼저 판단하고, AI에게 원인 가설, 추가 검증 데이터, 현장 테스트 항목까지 묻는 방식으로 질문을 설계해야 한다고 설명했다.
- 분석 결과를 원페이지 보고서로 바꿀 때는 SCQA 구조를 사용해 상황, 문제, 질문, 해법이 하나의 내러티브로 이어지게 해야 한다고 설명했다.
- 블로그형 문서, 랭킹 리포트, 인포그래픽, 슬라이드처럼 같은 분석을 다양한 형식으로 재가공할 수 있지만, AI 초안 뒤에는 항상 사람의 후편집과 맥락 보정이 필요하다고 강조했다.
- 수업 후반에는 LLM 위키, Obsidian, 메모리 부스트, 그래프 기반 연결 구조를 통해 과거 작업과 문서 맥락을 다시 불러오는 시스템을 시연했다.
- AI와 장기적으로 협업하려면 매번 같은 컨텍스트를 다시 넣는 방식이 아니라, 작업 흐름과 변화 이력을 기억하는 메모리 구조가 필요하다고 설명했다.

## 핵심 메시지
1. **좋은 분석은 좋은 데이터 구조에서 시작된다.**
사람 중심 서식과 AI 중심 구조는 다르며, AX 실무에서는 AI가 해석하기 쉬운 형태로 표를 정리하는 능력이 매우 중요하다.

2. **EDA는 통계 기법보다 질문 설계가 먼저다.**
이상치가 왜 생겼는지, 무엇을 더 확인해야 하는지, 어떤 추가 데이터가 필요한지를 묻는 방식이 실무 분석의 수준을 결정한다.

3. **원페이지 보고서는 숫자 요약이 아니라 설득 구조다.**
SCQA처럼 상황, 문제, 질문, 해법이 자연스럽게 연결되어야 경영진이나 실무 책임자가 바로 행동으로 옮길 수 있다.

4. **AI 활용은 한 번의 분석보다 기억 구조가 중요하다.**
LLM 위키와 메모리 시스템이 있어야 과거 문서, 설정, 작업 맥락을 이어받아 반복 업무를 더 빠르고 정확하게 수행할 수 있다.

## 기억해야 할 디테일
- Excel 시트는 병합 셀과 다중 헤더를 줄이고, 열별 속성이 분명한 구조로 정리하는 것이 좋다.
- 복잡한 표는 원본 파일보다 이미지나 정리된 중간 표로 전달하는 편이 AI 해석 정확도를 높일 수 있다.
- 공개 데이터셋은 연습용으로 유용하지만, 실무 데이터는 보안과 품질 관리 기준이 함께 따라와야 한다.
- 프로젝트 데이터 해석에서는 연도별 평균만 보지 말고 `Not Started`, `In Progress`, `Delayed` 같은 상태 구성을 함께 봐야 한다.
- 이상치 분석은 `삭제 여부`보다 `원인 가설`과 `추가 검증 계획`까지 연결해야 실무 가치가 생긴다.
- AI 초안은 빠르지만 최종 문서 품질은 사람의 후편집, 용어 통일, 조직 맥락 반영에서 완성된다.
- LLM 메모리 시스템은 AX가 중반 이후로 갈수록 중요해지며, 반복 컨텍스트 입력 비용을 줄이는 핵심 장치가 된다.

## 결론
4월 17일 수업은 단순한 Excel 분석 실습이 아니라, **좋은 데이터 구조를 만들고, 좋은 질문으로 분석하고, 그 결과를 의사결정용 보고서로 번역하며, 다시 장기 기억 구조에 축적하는 전체 AX 흐름**을 다룬 시간이었다. 핵심은 AI 자체보다 `데이터 구조`, `질문 구조`, `보고서 구조`, `메모리 구조`를 함께 설계하는 능력이 실무 AX 성과를 좌우한다는 점이다.
