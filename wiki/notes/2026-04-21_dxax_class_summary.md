# 2026-04-21 DXAX 수업 한페이지 요약

## 수업 전체 요약
이날 수업의 핵심은 **전날 배운 `n8n` 자동화 개념을 실제 설계 프레임워크와 데이터 구조 이해로 확장해, AI가 판단하는 워크플로우를 더 안정적으로 만드는 방법**을 익히는 것이었다. 초반에는 Trigger, Action, Condition, ESA 원칙, 실무자 중심 자동화라는 개념을 복습했고, 이어서 자동화 설계에서 가장 중요한 것이 `업무를 어떻게 쪼개고 어떤 형식의 데이터로 전달할지`라는 점을 강조했다. 중반부에는 `JSON` 구조와 key-value 개념을 집중적으로 설명하며, 자동화 노드 사이에서 오가는 데이터가 결국 JSON이므로 이를 읽고 해석할 수 있어야 디버깅과 설계가 가능하다고 정리했다. 후반부에는 이메일 분석기, 감정분석/분류 워크플로우, AI 라우터, 예외 처리, 실행 로그 추적, 문서화, 피어 리뷰, Notion API 연동까지 이어지는 실습으로 수업이 진행됐다.

## 무엇을 배웠나
- 전날 다뤘던 `n8n`, Zapier, Make 같은 자동화 도구를 다시 비교하며, 이번 과정의 핵심은 도구 이름보다 `AI + 워크플로우 + 데이터 구조`를 함께 설계하는 것이라는 점을 정리했다.
- 실무자가 AX를 직접 진행해야 하는 이유를 설명했다. 도메인 지식과 현업의 예외 상황은 실무자가 가장 잘 알고 있으므로, 개발자에게 전부 전달하는 대신 실무자가 직접 로우코드/노코드 도구로 자동화를 설계할 수 있어야 한다고 강조했다.
- 자동화 설계의 기본 프레임으로 `Trigger`, `Action`, `Condition`을 다시 정리했다. 언제 시작되는지, 무엇을 실행하는지, 어떤 조건에서 분기하는지가 자동화 구조의 뼈대라는 점을 설명했다.
- 성공적인 자동화를 위한 원칙으로 전날 소개한 `ESA(Eliminate, Standardize, Automate)`를 다시 연결했다. 먼저 불필요한 단계를 제거하고, 데이터를 표준화한 뒤, 마지막에 자동화해야 안정적인 워크플로우가 된다는 점을 반복해서 강조했다.
- 자동화 설계에서는 프롬프트 자체보다 `프로세스 설계`가 더 중요하다고 설명했다. AI에게 무엇을 하라고 시키기 전에 업무의 목표, 입력, 로직, 출력, 최종 목적지를 분해해야 한다는 점을 다뤘다.
- 중반부 핵심은 `JSON 이해`였다. n8n 노드 사이에서 이동하는 데이터가 대부분 JSON이므로, key-value 구조, 배열, 속성 경로, 필요한 필드 추출 방식을 이해해야 노드 출력값을 읽고 활용할 수 있다고 설명했다.
- “값(value)에 의미가 있고, 그 값을 꺼내려면 key를 정확히 지정해야 한다”는 관점으로 JSON을 읽는 법을 설명했고, 세션 ID나 특정 속성값을 다음 노드에서 참조하는 흐름을 사례로 보여줬다.
- AI 라우터 개념도 다뤘다. 단순 규칙 기반 스위치가 아니라, 애매한 자연어 입력이나 이메일 요청을 AI가 해석해 적절한 후속 경로로 분기시키는 로직이 자동화의 차별점이라는 점을 설명했다.
- `이메일 분석기(email analyzer)` 워크플로우를 만들며, 시스템 메시지에 “이메일 분류 전문가” 역할을 주고, 카테고리와 next action을 지정된 형식으로 반환하게 만드는 구조를 실습했다.
- 이메일 분류와 함께 감정분석(sentiment analysis) 워크플로우도 실습했다. positive / neutral / negative 같은 분류 결과를 시트나 데이터베이스로 보내는 과정을 다루며, 출력 형식이 맞지 않으면 downstream 노드가 깨질 수 있다는 점을 확인했다.
- 이 과정에서 `표준화가 안 되어 있으면 노드 간 연결이 쉽게 깨진다`는 것을 실제 오류 사례로 보여줬다. JSON 형식이 어긋나거나 필수 키가 빠지면 후속 노드가 정상 작동하지 않는다는 점을 확인했다.
- 후반부에는 `try-catch` 성격의 예외 처리 로직과 폴백 경로를 실습했다. 잘못된 API 키, 빈 데이터, 필수 필드 누락, API rate limit 같은 상황을 일부러 만들고, 어떻게 로그를 보고 우회/알림 경로를 설계할지 설명했다.
- 실행 로그를 읽으며 어느 노드에서 실패했는지, 어떤 데이터가 들어갔는지, 왜 출력이 없었는지 추적하는 디버깅 습관을 강조했다. 자동화는 성공 케이스보다 실패 케이스를 추적할 수 있어야 운영 가능하다는 맥락이었다.
- 자동화 흐름을 `목표-입력-로직-출력` 구조로 문서화하는 작업도 중요하게 다뤘다. 사람이 읽어도 이해되는 문서가 있어야 도구가 바뀌어도 업무 로직을 다시 이식할 수 있고, 더 나은 프레임워크로 옮기는 것도 쉬워진다고 설명했다.
- 문서화 후에는 피어 리뷰와 AI 리뷰를 함께 활용해 정확성, 누락 방지, 권한/API 제한, 에러 담당자 명시 여부를 점검해야 한다고 설명했다.
- 마지막 구간에서는 `Notion API` 연동 실습을 진행했다. 데이터베이스 ID 확인, 속성명 매핑, 커넥터 연결, 페이지/데이터베이스 조회, 코멘트 블록 처리 등을 보여주며 자동화 결과를 Notion DB에 적재하는 흐름을 시연했다.

## 핵심 메시지
1. **자동화의 품질은 AI 모델보다 설계 구조에 달려 있다.**  
Trigger, Action, Condition과 ESA 원칙을 명확히 설계하지 않으면 좋은 모델을 붙여도 불안정한 자동화가 된다.

2. **JSON을 읽을 수 있어야 자동화를 디버깅할 수 있다.**  
n8n에서 오가는 데이터는 결국 JSON이므로, key-value 구조와 필요한 속성 경로를 이해해야 에러 원인을 추적하고 후속 노드를 정확히 연결할 수 있다.

3. **AI의 역할은 단순 생성이 아니라 판단과 라우팅이다.**  
애매한 이메일, 자연어 요청, 감정 분류처럼 규칙만으로 처리하기 어려운 구간에서 AI가 라우터 역할을 할 때 자동화의 실전 가치가 커진다.

4. **운영 가능한 자동화는 예외 처리와 문서화가 함께 있어야 한다.**  
성공 경로만 만드는 것이 아니라 API 오류, 빈 데이터, 권한 문제, 포맷 오류를 대비하고, 그 구조를 문서로 남겨야 실제 조직 안에서 유지될 수 있다.

## 기억해야 할 디테일
- 자동화 설계는 `목표-입력-로직-출력-최종 목적지` 순서로 정리하면 구조가 선명해진다.
- JSON에서 필요한 값을 꺼내려면 key 이름과 경로를 정확히 알아야 하며, 이 기본기를 모르면 노드 연결이 자주 깨진다.
- 이메일 분석기처럼 카테고리와 next action을 명시적으로 반환하도록 프롬프트를 설계하면 후속 자동화가 쉬워진다.
- 감정분석, 분류, 라우팅 같은 AI 노드는 출력 스키마가 조금만 달라도 downstream에서 오류가 날 수 있으므로 표준화가 중요하다.
- try-catch, 폴백, 오류 알림 노드가 있어야 빈 데이터·잘못된 API 키·rate limit 같은 상황에서 전체 워크플로우가 멈추지 않는다.
- 실행 로그를 읽는 습관이 중요하다. 실패 원인과 입력 데이터를 못 보면 자동화는 운영 단계에서 금방 블랙박스가 된다.
- 문서화는 단순 기록이 아니라 다른 사람이 읽고 구조를 상상할 수 있게 만드는 작업이어야 한다.
- Notion 연동 시에는 데이터베이스 ID, 속성명, 커넥터 연결 상태, 페이지/코멘트 블록 위치까지 정확히 맞춰야 정상 동작한다.

## 결론
4월 21일 수업은 **자동화 도구를 쓰는 법을 넘어서, 자동화가 왜 실패하고 어떻게 안정적으로 운영 가능한 구조로 바꿀 수 있는지를 배우는 수업**이었다. 핵심은 AI 노드를 하나 추가하는 것이 아니라, JSON 구조를 이해하고, AI 판단 로직을 설계하고, 예외 처리와 문서화를 포함한 운영 구조를 갖추는 데 있었다. 결국 실전 AX는 “한 번 돌아가는 데모”보다 “에러가 나도 추적하고 수정하며 계속 운영할 수 있는 워크플로우”를 만드는 역량이라는 점이 반복해서 강조됐다.
