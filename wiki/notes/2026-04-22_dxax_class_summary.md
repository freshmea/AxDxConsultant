# 2026-04-22 DXAX 수업 한페이지 요약

## 수업 전체 요약
이번 수업의 핵심은 개인이 혼자 쓰는 자동화에서 한 걸음 더 나아가, 팀이 함께 일하는 협업 환경 안에서 AX를 어떻게 설계할 것인가였다. 전날 다룬 AI 라우터와 JSON 기반 워크플로우를 다시 점검하면서도, 단순히 “내 일을 자동화하는 법”보다 “여러 사람이 같은 맥락을 공유하고, 같은 데이터를 보고, 같은 프로세스로 움직이게 만드는 법”에 더 큰 비중을 뒀다. 단순한 `switch`나 `if` 분기 대신 AI 라우터를 쓰는 이유도 개인 편의성 때문이 아니라, 팀으로 들어오는 애매한 요청을 공통 기준으로 해석하고 다음 액션으로 연결하기 위해서라는 점을 강조했다.

또한 Obsidian 그래프, 메모리 레이어, Codex 검색, Notion 같은 협업 소프트웨어와 지식 관리 도구를 예로 들며, AX의 방향이 개인 생산성 툴을 넘어서 팀 운영용 SaaS와 협업형 시스템으로 확장되고 있음을 설명했다. 즉, AI는 혼자 쓰는 비서가 아니라 팀의 문맥을 기억하고, 자료를 정리하고, 부서 간 소통을 보조하는 공동 작업 인프라가 되어야 한다는 것이다. 후반부의 비정형 문서 전처리, 긴급 TFT 회의 시뮬레이션, 정부 사이트 데이터 수집 자동화 역시 모두 개인 자동화가 아니라 조직 단위의 정보 흐름과 협업 체계를 더 잘 굴리기 위한 사례로 연결됐다.

## 무엇을 배웠나
- 워크플로우 도구에서 노드 간 데이터가 어떻게 JSON으로 이동하는지 다시 점검했다.
- 단순 규칙 기반 분기와 AI 라우터의 차이를 비교하며, 의미 기반 분류의 장단점을 이해했다.
- 이메일, 불만 접수, 견적 문의, 스팸, 이커머스 CS 같은 사례에 AI 라우터를 확장하는 방식을 살펴봤다.
- 지금까지의 개인 작업 중심 자동화가 팀 작업 중심 AX로 어떻게 확장되는지 큰 흐름을 정리했다.
- Obsidian 그래프, 메모리 레이어, Codex 검색을 통해 짧은 질문만으로도 과거 문맥을 되살리는 협업형 지식 환경을 확인했다.
- Notion 페이지 구조와 하위 페이지 개념을 통해 업무 지식을 체계적으로 쌓고 팀이 함께 재사용하는 방식을 이해했다.
- 협업 소프트웨어와 SaaS는 단순 기록 도구가 아니라 팀의 문맥, 업무 상태, 후속 액션을 묶어 주는 운영 레이어라는 점을 배웠다.
- 종이 문서와 비정형 자료를 스캔, OCR, Markdown 변환을 거쳐 AI가 읽을 수 있는 지식베이스로 만드는 흐름을 배웠다.
- 에이전트 운영에는 토큰 비용이 들지만, 잘 설계된 에이전트는 사람의 반복 업무를 크게 줄일 수 있다는 운영 관점을 다뤘다.
- 긴급 TFT 킥오프 회의 시뮬레이션을 통해 AI가 감정을 걷어내고 데이터 중심 액션 플랜을 정리하는 팀 조정자 역할을 수행할 수 있음을 확인했다.
- 정부 지원사업 게시판 같은 웹페이지를 반복 수집하기 위해 HTML 요청, 반복, 페이지네이션, JSON 접근 로직을 구성하는 법을 살펴봤다.
- 필요한 추출 로직은 GPT를 이용해 JavaScript나 정규식 코드 형태로 빠르게 생성해 자동화에 붙이는 방식을 실습했다.

## 핵심 메시지
- AI 라우터는 `if`문보다 유연하지만, 그 유연함만큼 검증과 책임 구간을 별도로 설계해야 한다.
- 좋은 AX는 개인이 혼자 빠르게 일하는 데서 끝나지 않고, 팀이 같은 문맥과 프로세스를 공유하도록 만드는 데서 완성된다.
- 좋은 자동화는 프롬프트를 길게 쓰는 데서 나오지 않고, 지식베이스와 메모리 구조를 팀 차원에서 잘 쌓는 데서 나온다.
- AX의 시작은 멋진 모델보다 먼저, 현실의 비정형 자료를 기계가 읽을 수 있는 형태로 바꾸는 전처리 역량이다.
- AI는 단순 실행 도구를 넘어, 부서 간 갈등을 정리하고 액션 아이템을 구조화하는 협업 조정자 역할로도 확장될 수 있다.

## 기억해야 할 디테일
- 워크플로우 안에서는 거의 모든 데이터가 JSON 단위로 흐르므로 출력 포맷을 구조화하는 습관이 중요하다.
- AI 라우터는 분류 정확도를 높일 수 있지만, 오분류 가능성이 있으므로 사람 검토나 후속 확인 단계를 남겨야 한다.
- Codex와 메모리 레이어를 잘 연결하면 긴 배경설명 없이도 과거 문서와 맥락을 다시 불러올 수 있고, 팀 단위 협업에서도 같은 문맥을 공유할 수 있다.
- Obsidian, Notion 같은 협업형 소프트웨어는 AX에서 문서 저장소가 아니라 팀의 기억과 작업 상태를 유지하는 인프라로 작동한다.
- 실제 현장 문서는 디지털 문서보다 종이, 스캔본, 수기 자료가 더 많기 때문에 OCR과 Markdown 변환이 필수다.
- 반복 수집 자동화에서는 HTML 요청, 페이지 반복, 링크 추출, 상세 페이지 접근, 필드 파싱 순서가 중요하다.
- 데이터 추출용 JavaScript나 정규식은 AI에게 먼저 초안을 만들게 하고, 사람이 검증하면서 다듬는 방식이 효율적이다.
- 에이전트는 공짜가 아니며, 토큰 비용과 성능을 함께 고려해 운영해야 한다.

## 결론
2026년 4월 22일 수업은 개인 생산성 향상 중심의 AX에서 팀 협업 중심 AX로 시야를 넓히는 시간이었다. 핵심은 AI를 각자 따로 쓰는 도구로 두는 것이 아니라, 협업 소프트웨어와 연결해 팀의 문맥을 기억하고 업무를 이어 주는 공동 인프라로 키우는 것이다. 이를 위해서는 JSON 구조 이해, 지식베이스 구축, 협업형 SaaS 활용, 비정형 문서 전처리, 반복 수집 자동화 같은 기반 작업이 함께 갖춰져야 한다.
