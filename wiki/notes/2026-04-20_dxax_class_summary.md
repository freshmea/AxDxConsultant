# 2026-04-20 DXAX 수업 한페이지 요약

## 수업 전체 요약
이날 수업의 핵심은 **AI를 단순히 답변 생성 도구로 쓰는 수준을 넘어, `n8n` 기반 워크플로우와 데이터 구조를 이용해 실제 업무 자동화 체계를 설계하는 방법**을 익히는 것이었다. 오전에는 지난 2주 동안 다뤘던 Codex, GPT 웹, 스킬, 메모리 구조를 다시 정리하면서 왜 3주차부터는 `자동화`로 넘어가야 하는지를 설명했고, 전통 개발, 로우코드, 노코드, AX 자동화의 차이를 비교했다. 이어서 DB, 자동화 엔진, AI를 연결하면 “나만의 디지털 비서”를 만들 수 있다는 관점에서 `n8n` 아키텍처와 성공/실패 패턴을 설명했다. 오후에는 실제 `n8n` 화면을 보며 Trigger, Action, Condition, 데이터 테이블, 구글 시트 연동, 로그 확인, 템플릿 활용, 셀프 호스팅까지 이어지는 실습 중심 흐름으로 수업이 진행됐다.

## 무엇을 배웠나
- 지난 수업에서 사용한 Codex, GPT 웹, 스킬, 메모리 구조를 다시 복기하면서 웹형 GPT는 편하지만 반복 맥락 입력이 필요하고, Codex는 파일·도구·구조를 묶어 지속형 업무에 더 적합하다는 점을 정리했다.
- 스킬과 메모리 구조를 잘 설계하면 프롬프트를 매번 처음부터 길게 쓰지 않아도 되고, AI를 “똑똑한 부하직원”처럼 운영할 수 있다는 관점을 설명했다.
- 3주차의 주제는 단순 프롬프트 작성이 아니라 `자동화로 확장되는 AX`이며, 지금까지 익힌 요약·조사·문서화·위키·메모리 구조가 자동화의 기반이라는 점을 연결했다.
- 전통 개발, 로우코드, 노코드, AX 자동화를 비교하면서 개발 리드타임과 필요한 인력 구조가 어떻게 달라지는지 설명했다.
- `DB + 자동화 엔진 + AI`를 연결하면 정형 데이터 처리와 인지적 판단을 한 흐름 안에서 묶을 수 있고, 이 조합이 개인/팀 단위의 디지털 비서가 된다는 점을 설명했다.
- 자동화 실패 사례로는 너무 큰 범위로 시작하는 경우, 예외 처리를 빼먹는 경우, 유지보수 구조가 블랙박스가 되는 경우를 들었고, 따라서 작은 단위부터 성공을 쌓아야 한다고 강조했다.
- 성공적인 자동화를 위한 원칙으로 `ESA`를 제시했다. E는 제거(불필요 단계 제거), S는 표준화(시스템이 읽을 수 있는 데이터 형식 통일), A는 자동화(정돈된 프로세스 위에 도구를 얹는 것)라는 구조였다.
- 오후 실습에서는 `n8n`의 기본 인터페이스와 워크플로우 실행 구조를 살펴보며, 채팅 메시지 같은 입력이 Trigger가 되고 이후에 AI 노드와 후속 액션이 연결되는 흐름을 시연했다.
- 워크플로우를 운영할 때는 실행 로그를 반드시 보고, 특정 시점의 요청과 에러 원인을 다시 추적할 수 있어야 하며, 문제가 생기면 로직의 어느 지점이 잘못됐는지 역추적하는 습관이 중요하다고 설명했다.
- `Trigger`, `Action`, `Condition`을 자동화의 3대 구성요소로 정리했다. 하나의 Trigger에서 여러 Action을 병렬/직렬로 연결할 수 있고, 데이터를 이동·변환하거나 AI 프롬프트를 실행하고, 슬랙·구글시트·메신저 같은 채널로 결과를 보내는 식으로 확장할 수 있음을 보여줬다.
- 구글 시트에 숫자를 저장하거나, 데이터 테이블의 값을 누적시키는 예시를 통해 상태값을 외부 저장소에 남겨야 영속적인 자동화가 가능하다는 점을 설명했다.
- 실습 후반에는 피어 리뷰 방식으로 시나리오의 약한 지점을 점검했다. 추출할 데이터가 비어 있으면 어떻게 반응할지, 권한 문제로 앱 접근이 막히면 어떤 우회 경로를 둘지 같은 질문을 던지며 워크플로우를 보강했다.
- `n8n`의 템플릿, 어드민 패널, 대시보드, 실행량 제한 등을 함께 소개하며, 처음부터 전부 새로 만들기보다 검증된 템플릿을 참고해 빠르게 시작하는 방법을 안내했다.
- 마지막에는 `n8n`을 클라우드 서비스로 쓰는 방식과 직접 서버/도커로 셀프 호스팅하는 방식을 설명하며, 개인 PC나 별도 서버에 설치해 더 자유롭게 확장할 수 있다는 점을 소개했다.

## 핵심 메시지
1. **좋은 자동화는 좋은 프롬프트보다 좋은 프로세스 설계에서 시작된다.**  
AI를 붙이기 전에 먼저 업무를 쪼개고, 불필요 단계를 제거하고, 데이터 형식을 표준화해야 자동화가 안정적으로 동작한다.

2. **자동화는 작게 시작해서 넓혀야 한다.**  
처음부터 전사 ERP나 복잡한 전체 업무를 한 번에 묶으려 하면 실패 확률이 높다. 작은 성공 사례를 만들고 그다음에 범위를 넓히는 접근이 맞다.

3. **Trigger, Action, Condition을 이해하면 대부분의 업무 자동화는 구조화할 수 있다.**  
어디서 시작되고, 무엇을 실행하고, 어떤 조건에서 분기하는지를 정의하면 반복 업무는 상당 부분 워크플로우로 옮길 수 있다.

4. **AI 자동화의 경쟁력은 기억과 상태 관리에서 나온다.**  
로그, 데이터 테이블, 구글 시트, DB 같은 외부 저장소를 함께 써야 워크플로우가 일회성이 아니라 누적형 시스템으로 발전한다.

## 기억해야 할 디테일
- `n8n`은 단순 연결 도구가 아니라 Trigger, Action, Condition, AI 노드를 엮어 업무 절차를 실행 가능한 형태로 바꾸는 자동화 엔진이다.
- AI를 붙인다고 자동화가 성공하는 것이 아니라, 예외 처리와 로그 확인 구조를 같이 설계해야 운영 가능한 시스템이 된다.
- API로 주고받는 데이터는 가능한 한 JSON처럼 표준화된 형식으로 맞추는 것이 좋다.
- 하나의 Trigger에서 여러 후속 액션을 병렬로 실행할 수 있으므로, 데이터 저장과 알림 발송을 동시에 처리하는 식의 설계가 가능하다.
- 데이터 테이블이나 외부 시트에 값을 남기지 않으면 “누적 상태”가 사라져 영속적 자동화가 어렵다.
- 권한 문제, 빈 데이터, 외부 앱 장애 같은 실패 상황을 미리 질문하고 설계하는 피어 리뷰가 중요하다.
- 템플릿과 기존 사례를 먼저 참고하면 처음부터 모든 노드를 직접 설계하는 부담을 크게 줄일 수 있다.
- 셀프 호스팅은 설치 난도가 있지만 장기적으로는 더 자유로운 확장성과 제어권을 제공한다.

## 결론
4월 20일 수업은 **AI를 잘 쓰는 법을 넘어, AI가 실제 업무 안에서 반복적으로 일하게 만드는 구조를 설계하는 수업**이었다. 핵심은 `n8n`이라는 도구 자체보다, 업무를 Trigger-Action-Condition으로 분해하고, 표준화된 데이터와 외부 저장소를 활용해 예외 처리까지 포함한 자동화 시스템으로 바꾸는 사고방식이었다. 결국 AX의 실전 경쟁력은 “좋은 답변”보다 “지속적으로 굴러가는 워크플로우”를 만들 수 있는가에 달려 있다는 점이 반복해서 강조됐다.
